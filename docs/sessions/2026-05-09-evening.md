## 二、字典极致瘦身（A+B+C+D+E）

用户警句：「**世界词汇太多了，要节约不损耗，不能无限扩张**」

### A. synth 范围限缩
旧规则对所有 term 派生太激进（431 → 1071 corrections，2.5 倍率）。
新规则：
- 全大写 ≤6 字母 → 全派生（含字母替换 + 加空格 + 小写）
- 驼峰 → 仅派生拆词 1 个
- 普通词不派生
- **synth 排除英文 stop words**（防 `AND→and` 污染 bandwidth）

### B. terms 硬上限
- `polish.dict_max_terms = 800`（hint 用）
- `polish.dict_max_correction_terms = 300`（Levenshtein 用）
- terms 拆成 `terms` (SA hint) + `correctionTerms` (Levenshtein 子集)

### C. correct() timing log
`>5ms 或有 hits` 时 log，便于持续观察

### D. Levenshtein 长度桶
`asciiCorrectionByLength: [Int: [String]]`，token 长 N 时只扫 buckets[N-maxDist...N+maxDist]

### E. hint vs correction 拆分
两个 cap 独立调，前者注入 SA 可放宽，后者跑全表必须收紧

**结果**：
| 指标 | Before | After |
|---|---|---|
| sortedErrorKeys | 1118 | 348 |
| synth | 1071 | 296（-72%）|
| Levenshtein 桶数 | — | 31 |
| correct() 单次耗时 | 1-18ms | < 20ms 稳定 |

## 三、4 个误纠 bug 修复

| Bug | 现象 | 根因 | 修法 |
|---|---|---|---|
| **A. Cascading** | `banwise → bandwidth` 后被 `and→AND` 二次替换成 `bANDwidth` | L1 多遍替换，已替换部分被后续 err 触发 | single-pass left-to-right + 长词优先；`errKeysByFirstChar` 索引 |
| **B. AI→API 误纠** | `AI` 被 Levenshtein 拉成 `API` (d=1) | Levenshtein 没白名单 | 内置 ~500 高频英文词集合；**砍 NSSpellChecker** |
| **NSSpellChecker 死路** | 加白名单后 correct 飙到 193ms | NSSpellChecker 首次冷启动 200ms、单次 5-10ms | 改内置 `commonEnglishWords` set，性能 193ms→16ms |
| **C. idea→IDEa** | `ide → IDE` 在 `idea` 中误替换 | L1 没 word boundary | err 自身字符是 ASCII word char 时，左右不能是 word char |
| **NumberNormalizer 量词误转** | "我有一种想法" → "我有1种想法" | 单字 number char 99% 是量词 | tokenize 时 span < 2 跳过 |
| **FillerRemover 数字误压** | "30000000" → "3000" | compressRepeats 把 "00" 当中文重复块压了 | 跳过任何含 ASCII 字符的 block |

## 七、FR5 FillerRemover

新建 `FillerRemover.swift`：
- 删 boundary 单字 filler（嗯/呃/啊/哦/哎/唉/呵/嘿）
- 保留组合词（嗯哼 / 哎呀 / 嘿嘿 / 哦哦 / ...）
- 重复词压缩：`(.{1,3})\1{2,}` → 保留 1 次（"是的是的是的"→"是的"）
- 跳过 ASCII block（防数字 / 英文占位被压）
- 孤儿标点 cleanup（"，  ，"→"，"，开头空白+标点删）

集成位置：`VoicePipeline` 在 dict.correct 之后、PunctuationNormalizer 之前。

mixed-corpus 实测：删 13 chars（"呃" / "我们×3" / "是的×3"）。

## 八、FR6 NumberNormalizer L1-L4（ITN 雏形）

用户 reframe："**这是数字录入的通用问题**，不是补丁。"

业界叫 ITN（Inverse Text Normalization），主流 WFST + NN（飞书 / 讯飞 / Google）。LLM 端侧 240ms 上限不可行，规则版唯一选择。

新建 `NumberNormalizer.swift`：
- **L1** 单字符：零一二...九 → 0-9
- **L2** 复合：`一千二百三十四` → 1234（unit-based 状态机）
- **L3** 量级：`三千万` → 30000000，`一亿三千万` → 130000000
- **L4** 小数：`三点一四` → 3.14

**算法**：tokenize 找 number span → unit-based parse OR digit-by-digit 选其一 → 输出阿拉伯。

**实测**：
| Input | Output |
|---|---|
| 三千万 | 30000000 |
| 一亿三千万 | 130000000 |
| 二零二四年 | 2024年 |
| 一千二百三十四 | 1234 |
| 三点一四 | 3.14 |
| 我有一种想法 | 我有一种想法（"一" 量词保留）|
| 买了两个苹果 | 买了两个苹果（"两" 量词保留）|

**不解决**：截断恢复（`024年` → `2024年`，需要上下文）/ 上下文敏感量级补全（`100 token` → `100k token`）/ 音译错（`00造` → `100k`）— 这些都需要 LLM。
