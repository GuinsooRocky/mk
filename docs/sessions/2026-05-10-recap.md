## 三、流式探索两条死路

### 3.1 死路 #3 — B-prime（finalized-segment lookahead 流式）

**假设**：SA 会在说话过程中陆续 emit `isFinal=true` 段，每段 commit 一次，松手时只剩 tail。

**实测打脸**：3 次测试全部 `streaming=false`，SA zh-CN **完全不在说话过程中 emit isFinal**——所有 final 段都积压到 `finalize()` 后 60ms 内一次性喷出。

```
record 11.94s → SA 全程 0 个 Final segment 日志
release → finalize 60ms → 6 段一次性喷
```

→ 撤掉 ~150 行 B-prime 代码。

### 3.2 死路 #4 — Volatile-based 真流式

**假设**：SA 不给 isFinal 中段，但给 volatile，可以基于 volatile 做 backspace+repaste 流式。

**实测打脸**：
- SA volatile 也是延迟 11+ 秒才 burst 出来（log 实证：录音 11.79s 才出第一个 1 字 volatile）
- cc 终端 raw mode 下 **backspace 不可靠**：streaming commit 贴 raw → finalize backspace 撤不掉 → raw + corrected 在 cc 输入框里**并存**
  - 实例："呃 ，现在是不是流失啊...流失注入" + "现在是不是流式...流式注入" 两段都留下

→ 又撤回。

## 四、活路 — 主动 chunk 流式（V1）

用户重新定义"真流式"：**说 ≥2s 后，把前 2s 的文字输出**——这是分块流式（chunked streaming），不是字符级。

**设计**：每 chunkInterval（默认 2000ms）主动 finalize 当前 SA + 重启新 SA，captureDelegate 通过 `swapInputBuilder(to:)` 原子换 inputBuilder（NSLock 保护），音频流不停。

```
flushChunk() 时序:
  1. 创建新 SA 完成 start
  2. captureDelegate.swapInputBuilder(to: newBuilder)  ← 原子切换音频路由
  3. 老 inputBuilder.finish() + 老 analyzer.finalizeAndFinishThroughEndOfInput()
  4. 等老 resultTask drain（100ms 兜底）
  5. 拿 finalizedText + volatileText → onFinalChunk(cumulative)
```

**新文件**：实际是修改 `VoiceSession.swift` + `AudioCaptureDelegate` 加 `swapInputBuilder` + 新 `flushChunk()`。

**关键 bug 修复（迭代 2）**：第一版同时挂了 `onFinalChunk` 和 `onPartialResult`，导致 chunk swap 后新 SA 把积压音频快速 emit volatile，触发字符级 ⌘V 风暴（21ms 内 7 次 paste），iTerm 卡死。

→ 删掉 onPartialResult 路径，只保留 onFinalChunk。

**关键 UX 修复（迭代 3）**：第二版 chunk 贴 raw 文本，松手时 finalize 用 corrected 全文 → 与 streamed raw 几乎无 common prefix → backspace=72 + 重贴 71 字（整段闪烁）。

→ 每 chunk 提前过 `pipeline.correctText`，让用户直接看到纠错后的字（"流失"实时变"流式"），finalize 的 diff 几乎为 0。

**实测最终版**：21s 录音 → 10 个 chunks，每个 ~80ms 处理时间，单次 paste 一坨字，无字符风暴，无 iTerm 卡死。

## 五、落字延迟从 700ms 砍到 150ms（不靠流式）

意外副线收益。原 `VoiceSession.stop()` 里有：

```swift
try? await Task.sleep(for: .milliseconds(500))  // 兜底等 resultTask drain
```

**实测**：finalize() 返回那一刻 finalizedText 就齐了，500ms sleep 是纯空转。

**修法**：换成 `await resultTask.value`（精确等 stream 关闭 + drain 完成）+ 100ms 兜底超时防极端 race。

```
Before: stop_finalize=629ms (其中 500ms 是空转 sleep)
After:  stop_finalize=150ms (drain+cancel took 0.000s)
```

→ §2.3 红线 1 (≤220ms) **达标**。这条是今晚最值钱的发现。

## 主线：算法 A + 学习模式 V1 + WAV 重处理 + 批量 ROI 优化

### 算法 A：拼音哈希模糊匹配（CorrectionDictionary L3）

`CFStringTransform(kCFStringTransformMandarinLatin)` 把字典里所有 CJK term 转成不带声调的拼音字符串，建反向索引 `[pinyin → 正字]`。`correct()` 加 Layer 3：扫文本中 2-4 字 CJK 子串 → 算拼音 → 查表 → 命中替换。

**实测**：dict 里 65 个中文 term 入了索引，0 重复；说"流市/流氏/留时/柳市..." 任何同音字变体 → 全部归 "流式"。**消灭手工堆变体的 whack-a-mole**。

```
Total: hint=717 correction=300 errKeys=464 (manual=169) pinyin=65(0 dup)
```

### 学习模式 V1（CorrectionCapture 重建）

注入后 30s 调 `AXUIElementCopyAttributeValue` 读目标 app 的 focused text → CFStringTokenizer 中文分词 → LCS 对齐 → 抽 (oldToken, newToken) pair → 严格过滤（长度 2-5 字 + Levenshtein ≤2 + 长度比 0.5-2x）→ 调 `DictionaryLearner.learn` 自动入字典。

**默认开**，菜单 toggle 可关。配置 `polish.learning_enabled / learning_window_sec`。

### chunk 流式 V1 + 飞书风格修复

主动每 2s `flushChunk()`：创建新 SA pipeline + `swapInputBuilder(to:)` 原子换 inputBuilder（NSLock 保护）+ 老 SA finalize → 拿累计文本 → 喂 streamingInjector。

**3 次迭代修 bug**：
1. v1 + onPartialResult → 字符级 ⌘V 风暴 21ms 内 7 次 paste，iTerm 卡死
2. v2 chunks 走 raw 但 punct 把上次「。」改成「，」backspace 闪烁
3. v3 **飞书思路**：chunks 期间纯 raw（不跑 correctText），松手时 finalize 一次性大刷新；strip chunk 末尾 SA 假句号

**核心设计原则**（用户原话）："2 秒不是用户停顿，是软件切窗时间 → SA 在 chunk 边界加的「。」百分百是假句号，必须 strip"

### WAV 整段重处理（修 chunk 边界切碎）

实测发现 chunk 边界会切碎 token：
```
commit#10: +"1.95秒到两 2:"  ← 数字"1.11"被切到下个 chunk
commit#11: +"一点一一无秒"   ← 上 chunk 尾巴漂过来
```

**修法**：stop 时用 WAV 整段（连续音频）跑一次纯净 SA pass，作为 ground truth 替代 cumulativeChunkText。新 `VoiceSession.transcribeFromFile(at:)` 静态方法用 `analyzer.start(inputAudioFile:finishAfterFile:)` API。

**代价**：松手后多等 0.5-3 秒（M5 上 RT factor ~0.05-0.1）；换准确度。

## 关键设计原则汇总（今天）

1. **chunk 边界 ≠ 用户语义边界**（软件切窗 vs 自然停顿）→ 所有 chunk 边界的 SA 句末特征都该过滤
2. **流式期间贴 raw，松手时一次性大刷新**（飞书思路）→ 不在 chunk 里跑 correctText 避免 backspace 闪烁
3. **WAV 整段重处理是修 chunk 切碎的唯一干净方案** → 因为 audio 是连续录的，用 file-based SA 重跑能拿 ground truth
4. **算法 > 字典** → 拼音哈希消灭手工加同音字变体，未来字典只放正字一条
