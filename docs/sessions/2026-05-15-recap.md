# 2026-05-15 — 微信热键拦截 + 标点括号智能化（4 类括号 + 同音字鲁棒）

> 主线：解决两个长期烦人的体验问题
> 1. MK 右 Option 与微信"按住说话"撞键，在微信里 MK 把语音同步注入聊天框
> 2. 念"右括号"打出来字面三个字；念"中括号"出 `(中)` 残骸；念"诱惑号"（ASR 听错版本）完全救不回来

## 一、微信热键拦截（App Blocklist）

### 问题
- MK PTT 用右 Option（modifier-only）
- 微信 macOS 客户端"按住说话"默认也是 Option
- 双方都吃下按键 → 微信开始录音的同时 MK 把转写文本流式写入微信输入框

### 修法
**配置驱动的 app blocklist**，在 `GlobalHotKey` 回调入口短路：

```swift
// 在 handleFlags / handleKeyDown 的 press 边沿先调
if isFrontmostBlocked(cfg.appBlocklist) { return }
```

`NSWorkspace.shared.frontmostApplication` 是 main-thread-only，CGEventTap callback 在非 main thread，所以：
- start() 时订阅 `NSWorkspace.didActivateApplicationNotification`
- main thread 上更新 `cachedFrontmostBundle: String?`（nonisolated(unsafe)）
- callback 直接读这个缓存变量

### 决策：只拦 press 不拦 release
press 拦截 → 不入 stack；release 不看 blocklist → 保证已起来的 PTT 有配对的 release，避免按住时切窗造成 isPressed 状态泄漏。

### 默认值
`HotKeyConfig.default.appBlocklist = ["com.tencent.xinWeChat"]`

`RuntimeConfig` 首启 defaults 同步加，老用户 `~/.mk/config.json` 手补 `app_blocklist` 字段即可热生效。

### Settings GUI 兼容
`HotKeyRecorder` 录入新键时只产 `keyCode/modifiers/displayName`，`ViewModel.save()` 用新增的 `with(appBlocklist:)` 把旧 blocklist 合并回去——GUI 改键不丢拦截表。

## 二、标点括号智能化

### 起点
原来的 `PunctuationNormalizer` 用 strict isolated token 边界规则：
```swift
("左括号", "（"), ("右括号", "）"), ("括号", "（")
```
问题：
- isolated token 边界要求"括号"前后是非 letter 字符
- 中文连续语流里 "我说右括号" 的"括号"前后都是 CJK letter → 边界永远不命中 → 直接打字面字
- 即使命中也是**全角**，不适合写代码

### 新算法：smartParens（一个函数搞定）

设计目标——**只要 ASR 识别出"号"字相关 token，都尽量救回来**。
完整决策树：

```
扫到 "X 号"（中间可容忍空格/标点等 filler）
  X 必须 ∈ kuoChars: {括, 国, 火, 惑, 扩, 后, 阔, 廓}
    ↑ "括"(kuò) 的常见 ASR 同音/近音误识别
  ↓
回看 prev1（跳 filler 找第一个实质字符）
  prev1 ∈ modifierMap: {中,方 → []; 大,花 → {}; 尖 → <>}
    YES → kind = 对应类型，吞 modifier，再回看 prev2 找 left/right 前缀
    NO  → kind = 圆括号 (默认)，直接看 prev1 是不是 left/right 前缀
  ↓
left 前缀: {左}
right 前缀: {右, 又, 诱, 幼, 佑, 釉, 反}
            ↑ 右(yòu)的常见 ASR 同音误识别
  ↓
决定 shouldOpen:
  - leftPrefix    → 强制开
  - rightPrefix   → 强制闭
  - 都没有        → 看栈空 / 非空（栈记录已存在的 (/[/{/< 字符）
  ↓
输出: kind.opener 或 kind.closer
吞掉范围: [最远的命中字符 ... 号]
```

### 处理样本

| ASR 实际输出 | 旧版结果 | 新版结果 |
|---|---|---|
| `左括号 内容 右括号` | `（内容）` 全角 | `(内容)` 半角 |
| `左中括号 a 右中括号` | `（中a右中）` 残骸 | `[a]` |
| `左大括号 b 右大括号` | `（大b右大）` 残骸 | `{b}` |
| `又括号` (右被听成又) | 字面三字 | `)` |
| `诱惑号` (右括被听成诱惑) | 字面三字 | `)` |
| `中国号` (中括被听成中国) | 字面三字 | `[` 或 `]` |
| `右 括号` (中间空格) | 字面 | `)` |
| `右。括号` (中间句号) | 字面 | `)` |

### 字典 hint 补强
`~/.mk/correction-dictionary.txt` 加各种括号关键字（`括号 / 左括号 / 右括号 / 中括号 / 方括号 / 大括号 / 花括号 / 尖括号 / 左中括号 / 右中括号 / ...`）。让 SA 把 contextualStrings 里的这些词当 hint，**从源头降低**"括"被听成"惑/国/火"的概率——算法兜底 + 字典源头双层防御。

## 三、当天 bug 复盘

### 低级失误：guard 早退漏字
加完 kuoChars 同音字识别后，忘改 smartParens 入口的 guard：
```swift
guard text.contains("括") else { return text }  // BUG: 同音字识别完全失效
```
"中国号""诱惑号"等不含"括"字的全部走 early return。修复：
```swift
guard text.contains(where: { kuoChars.contains($0) }), text.contains("号") else { return text }
```

### Index invalidation 教训
v1 版 pairCloseBrackets 用 `String.Index` + `replaceSubrange`，mutation 后 index UB。新版统一改 `[Character]` 数组操作 + 整数索引，安全。
