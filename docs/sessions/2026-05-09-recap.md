## 五、C2 反向纠错（音译 → 正字）

字典 .txt 加 `|` 分隔语法：`正字 | 错音1 | 错音2`，
`CorrectionDictionary.corrections` 存 `[错音 → 正字]` map，
`VoicePipeline` 在 L1 后扫描文本做**精确字符串替换**，长错音优先匹配。

**实战**：依赖精确字符串匹配 → SA 每次音译变体不同（"Plus I can" vs "plus I cn"），覆盖率有限。corrections 基础设施保留，C5/C1/C4 都能复用。

## 六、C1 alternatives — 探明走不通

- `VoiceSession.reportingOptions += .alternativeTranscriptions` + final segment 输出 alternatives 日志
- **数据观察**：zh-CN locale 下大部分 segment "no alts"；有 alts 时**都是同样错的音译变体**（"swit fut UI" 的 alt 是 "sw克"），**没有英文原文**
- **根因**：zh-CN LM 推理空间不含英文 token，SwiftUI 这种英文词从来不在候选里
- **结论**：rescue 算法不实施；flag 留着供 server 蒸馏未来用

## 七、D1 en-US locale 实验 — 探明走不通

- 临时硬编码 `Locale(identifier: "en-US")`
- 用 `MK --test-alternatives <wav> --locale en-US` 跑现有中文 wav
- **结果**：en-US 模型把中文音频音译成英文短词「sorry / for that / are, I, / I, I,」
- **结论**：除非整句**全英文**说话，否则不能切 en-US；SA 不支持 session 内切换 locale → 撤回

## 八、C5 派生 + Levenshtein 模糊兜底（today's main work）

**用户洞察**：手补 SVG → SVJ/SAG/S A G 这种太累，应该有算法自动派生。

### C5 派生（加载时一次性，零运行时开销）

三条规则：
- **全大写缩写**（≤6 字母）：字母间空格 + 字母替换（V→A/J/F/B 等）。confusion 表内置 V/G/B/M/S/Z/X 7 个
- **驼峰**："SwiftUI" → "Swift UI" / "swift ui"
- **保底**：全小写

典型 100 词字典派生 ~500 corrections，加载 < 50ms。

### Levenshtein 模糊兜底（运行时，layer 1 未命中时触发）

经典 DP 实现（O(m×n)），阈值 `max(1, token.count/4)`。
长度差 > 阈值的 term 早 reject（O(1)）。仅对 ASCII letter token 启用。
实战 500 corrections + 50 字符文本 < 1ms。

### 实战边界

✓ 单字符差异（SVJ → SVG、Tlwind → Tailwind）
✓ 大小写/拆词差异（Swift UI → SwiftUI）
✗ 音译解构（"swit fut UI" 三 token，每个距 SwiftUI 都 > 阈值）
  → 这种结构性差异只能靠 C4 LLM 后处理

### 行业地位反思

用户问"飞书用的什么算法"。坦诚回答：
- 飞书 = 自研 Conformer/Transformer ASR + 自家训练数据 + 专门团队
- 我们 = Apple SA 端侧免费模型 + 应用层补救
- 两条路完全不同，**不能复刻飞书**
- 应用层的行业 standard：Levenshtein（教科书级）、音素哈希（Soundex/Metaphone）、LM rescoring（C4）

C5 是项目自定义启发式，Levenshtein 是 standard。两者组合是**实用主义版本**。

### ⚠️ 之前推断的"100 上限"是错的，实测推翻

我之前推断"SA 上限 100 是性能约束"——**错了**。用 `MK --test-context-capacity` 实测：

| 数量 | 0 | 50 | 100 | 500 | 1000 | 5000 |
|---|---|---|---|---|---|---|
| 转写耗时 | 210ms | 80ms | 90ms | 90ms | 90ms | 90ms |

- **5000 词跟 100 词延迟一样**——SA 内部用 O(1)/O(log n) 索引（哈希/trie）
- **0 词反而比有 hint 慢 130ms**——hint 收窄推理空间反而提速
- Apple 文档"≤100"是**效果建议**（hint 太多可能稀释准确率）**不是性能约束**

**改：`maxContextualStrings = 100 → 1000`**，让 codebase 字典全部注入 SA。
准确率影响待后续专门实测验证。

## 十二、给"未来的我"的 cheatsheet

- **看字典加载状态**：MK 启动 log 直接看 `Total: N terms + M corrections (manual=X, synth=Y) from K files`
- **看 correct 命中**：`tail -f ~/.we/debug.log | grep "Dict.*correct:"`，行格式 `=tok→正字` 是 Layer 1 精确，`~tok→正字(d=N)` 是 Layer 2 Levenshtein
- **看 alternatives**：`Voice Final segment: ... (N words, M alts)` + `alt[0]/alt[1]/...`
- **重扫 codebase 字典**：
  ```bash
  ~/Desktop/my-code/mike/client/scripts/scan-codebase.py --top 300 --min-freq 3 \\
    --out ~/.we/correction-dictionary-codebase.txt \\
    ~/Desktop/my-code ~/Desktop/cmm
  ```
- **重启 MK 标准流**：
  ```bash
  cd ~/Desktop/my-code/mike/client
  swift build && cp .build/debug/MK .build/MK.app/Contents/MacOS/MK && \\
    /usr/bin/find .build/MK.app -name "*.cstemp" -delete && \\
    codesign --force --deep --sign "MK Development" .build/MK.app && \\
    killall MK; sleep 2; open .build/MK.app
  ```
- **codesign 卡住**：keychain 密码框可能藏在隐藏窗口，点"始终允许"+ 输密码后下次不会再卡
- **`code has no resources`**：上次 codesign 没 `--deep` 或留了 `.cstemp` 残骸；用 `find ... -name "*.cstemp" -delete` + `codesign --force --deep --sign ...` 重签
- **改 ~/.we/config.json 之后**：rebuild 比期望 file watcher 自动热更靠谱
- **加错音**：拿日志 `Voice Final segment:` 的实际错文对照预期，加 `正字 | 错音` 到 `~/.we/correction-dictionary.txt`，下一次按热键即生效（hot reload）
